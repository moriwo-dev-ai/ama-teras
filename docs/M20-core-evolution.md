# M20: コア自己進化(提案制・保護領域つき)

目的: 自己進化の対象を「ツール生成のみ」から「本体コア(UI・エージェントループ・実行戦略など)」へ拡張する。
ただし**承認機構そのものは絶対に触らせない**。全変更は PR 形式で人間承認を必須とする。

**このマイルストーンは他と独立で、M17〜M19が完了・実機検証された後に単独で慎重に実施すること。**
既存の進化パイプライン(worktree / ゲート / 昇格 / ロールバック)を土台に、保護と検証を強化する。

## 大原則(corrigibility = 是正可能性を手放さない)

1. **保護領域は進化ジョブから変更禁止**。承認ダイアログにすら出さず、パイプラインが diff 検査で
   機械的に即拒否(reject)する。保護領域は以下:
   - `src/main/evolution/`(進化マネージャ・ジョブ・worktree・ゲート・昇格・ロールバック)
   - スーパーバイザー(再起動・健全性チェック)
   - 承認ブローカー + 承認UI(ApprovalBroker / ApprovalDialog / 関連IPC)
   - secrets 関連(secrets.ts / Local State 移行 / 鍵素材)
   - `CLAUDE.md` と **保護領域リストの定義ファイル自体**(自分で保護を外せないように)
2. **保護領域の判定は保護領域内で行う**(`src/main/evolution/protected.ts`)。この判定ロジックを
   進化ジョブが書き換えられないことを、保護対象に含める+ guardrails テストで固定
3. **コア変更は必ず人間承認**。ツール進化で自動昇格設定にしていても、コア変更カテゴリは
   自動昇格の対象外(常に diff 全文レビュー→承認)
4. **段階的権限**: 進化ジョブに与える書き込み許可を段階化する(下記スコープ)

## 進化スコープの段階(evolution write scope)

`request_capability` / コア進化要求に `scope` を持たせる:

| scope | 書き込み許可 | 承認 | 昇格後 |
|---|---|---|---|
| `tool`(従来) | `src/main/tools/plugins/` のみ | ツール設定に従う | ホットリロード |
| `renderer` | `src/renderer/`(UI)。保護UI(承認ダイアログ)除く | 常に人間承認 | 再ビルド+再起動 |
| `core` | `src/main/` の保護領域以外(agent loop / providers / core / mcp 等) | 常に人間承認 | 再起動+健全性チェック |

- どの scope でも保護領域への差分が1行でもあれば**即 reject**(検証以前の最初のゲート)
- scope は要求時に明示 or マネージャが diff から推定。推定と宣言が食い違えば厳しい側を採用

## 検証ゲート(コア変更は強化)

順序(最初の2つが安全ゲート):
1. **保護領域トリップワイヤ**: diff が保護領域に触れたら即 reject(最優先・無条件)
2. **危険検出**: child_process / ネットワーク / eval 等を diff から検出し、昇格ダイアログで明示警告
3. typecheck(3構成)
4. vitest 全合格
5. **build 成功**(コア変更はビルドが通ること必須)
6. **フルアプリ・スモーク起動**: `AMATERAS_SMOKE`(既存)で headless 起動し、
   起動が完了し基本IPCが応答すること(=アプリが壊れて起動不能にならない保証)
7. M19 の品質レビュー(任意・ONなら planner が自己変更を採点)

いずれか不合格 → feedback 付き再生成 or 失敗終了。合格 → 昇格承認へ。

## 昇格と安全な再起動・自動復旧

- 昇格 = main へ merge --no-ff + タグ `evolve/N`(既存)。コア/renderer は再ビルドが必要
- **再起動はスーパーバイザーが担当**(保護領域):
  1. 昇格前の稼働バージョンのタグを記録
  2. 再ビルド → 再起動
  3. **健全性チェック**(起動完了・IPC応答・主要画面レンダリング)
  4. 失敗 → 直前タグへ自動 revert して再起動(**セーフモード起動**)。ユーザーに通知
- 健全性チェックが連続失敗する場合は「進化機能を一時停止して素の状態で起動」するセーフモードを用意
  (承認機構と復旧経路だけは常に生きている状態を保証)

## UI

- 進化パネルに scope バッジ(tool / renderer / core)と「保護領域に触れたため拒否」表示
- コア変更の昇格ダイアログ: diff 全文 + 影響範囲(再起動の要否)+ 危険警告 + 「これは本体の
  コア変更です」の強調。承認は二段確認(チェック+ボタン)
- ロールバック履歴(evolve タグ)と「1つ前へ戻す」ボタン

## テスト(安全性の固定が主目的)

- **保護領域トリップワイヤ**: 保護対象の各パスへの差分を含むジョブが必ず reject されること
  (evolution/ / approval / supervisor / secrets / CLAUDE.md / protected.ts 自身)
- protected.ts を書き換えようとする差分が reject されること(自己保護の自己参照)
- scope 別の書き込み許可(tool は plugins のみ / renderer は src/renderer / core は保護外の src/main)
- スモーク起動ゲート: 起動不能になる差分が昇格前に落ちること
- 健全性チェック失敗 → 自動 revert → セーフモード起動
- コア変更は自動昇格設定でも人間承認が要ること
- 既存のツール進化(scope=tool)が従来どおり動くこと(回帰)

## 受け入れ基準

- 既存全テスト+新規合格 / typecheck / build
- 実証: `renderer` scope で小さなUI改善(例: ヘッダにボタン追加)をAIに提案させ、
  ゲート通過→承認→再ビルド→反映、まで通ること
- 実証: 保護領域(例: 承認ダイアログ)を変更しようとする要求が**必ず reject** されること(安全性の証明)
- セーフモード復旧が実際に効くこと(わざと起動失敗する差分での復旧デモ)

## やらないこと(このマイルストーンの外)

- 自動昇格(コア変更は常に人間承認)
- 保護領域の縮小・撤廃
- 分散配布/ネットワーク共有(別構想。まず単体で有用性を蓄積してから)
