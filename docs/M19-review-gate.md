# M19: 品質レビュー・ゲート(planner採点 → 差し戻しループ)

目的: 「動くか」だけでなく「質が高いか」を評価し、低ければ自動でテコ入れする。
M18(役割分担)の自然な発展。planner帯(高モデル)が成果物を採点し、閾値未満なら
具体的な指摘つきで worker に差し戻す。

## 発動: マイルストーンごと

- トリガー = 計画ファイル(AMATERAS_PLAN.md)の項目(またはグループ)が完了扱いになった時
  (`- [ ]` → `- [x]` へ更新された瞬間)
- 計画を使っていない短いタスクでは、エージェントループ完了時に1回だけ発動(=「完成時レビュー」に縮退)
- 進化ジョブには適用しない(既存4ゲートが担当。guardrailsで固定)

## 採点(4軸・plannerが厳格レビュー)

planner に「辛口レビュアー」として、そのマイルストーンの成果物を4軸で採点させる:

1. **コード品質**: 可読性・設計・重複・命名・エラー処理・エッジケース考慮
2. **見た目/UX**: Web/GUI成果物のとき、M14 の screenshot でUIを自己確認し、
   レイアウト崩れ・視認性・使いやすさを採点(UIが無いタスクではこの軸をスキップ)
3. **要件の達成度**: 最初のユーザー依頼 + AMATERAS_PLAN.md の該当項目を漏れなく満たしているか
4. **テストの質**: テストが通るだけでなく、カバレッジ・意味のあるケース・異常系を見ているか

- 各軸を 1〜5 で採点(UI無しでスキップした軸は対象外)。総合が閾値(既定 4.0)未満なら不合格
- レビューは**具体的**であること: 「ファイル・箇所・何が問題・どう直すか」を列挙(抽象的な指摘を禁止)
- planner のプロンプトに「自分が書いた/レビューした前提を持たず、第三者として厳しく見る」を明記

## 差し戻しループ

- 不合格 → 指摘リストを worker に渡して修正(M18のworker帯で実行)→ 再レビュー
- **上限 maxRoundsPerMilestone(既定 2)**。上限に達しても閾値未満なら、
  残課題をユーザーに提示して先へ進む(無限ループ防止)。自律モードOFF時はここで承認を仰ぐ
- worker が修正に2回失敗したら M18 のエスカレーション(escalation帯へ格上げ)と連動
- 各ラウンドの採点・指摘・差分を**レビューカード**としてチャットに表示 + audit.jsonl に記録
- レビュー履歴は右ペインに「レビュー」タブ(任意)or 計画パネルに採点バッジ

## 設定(Settings「品質レビュー」節)

```ts
reviewGate?: {
  enabled: boolean;          // 既定 false(ONで従来から挙動が変わるため)
  threshold: number;         // 既定 4.0(1〜5)
  maxRoundsPerMilestone: number; // 既定 2
  axes: { code: boolean; ux: boolean; requirements: boolean; tests: boolean }; // 既定 全ON
};
```

- プリセット: 「しっかり(閾値4.5/3周)」「標準(4.0/2周)」「軽め(3.5/1周)」
- レビューは planner 帯(高モデル)を使うためコスト増。マイルストーン単位に絞ることでバランスを取る旨をUIに注記
- OFF(既定)なら完全に従来挙動

## M16〜M18 との関係

- レビューは planner 帯で実行(M18 の planner モデル)。M18 OFF時は本体モデルで実行
- 修正実行は worker 帯 → 詰まれば escalation(M18)
- 課金エラー時は M16 フォールバックに乗る
- screenshot 軸は M14 の screenshot ツール(localhost自動)を使用

## テスト

- enabled=false で完全に従来挙動(既存テスト不変)
- モックplannerで: 高スコア→そのまま通過 / 低スコア→指摘リスト生成+worker再実行が走る
- maxRounds 到達で停止しユーザーへ残課題提示(自律OFFなら承認要求)
- UI無しタスクで ux 軸がスキップされる / UIありで screenshot が呼ばれる
- 進化ジョブに非波及(guardrails)
- レビュー結果が audit.jsonl に記録される

## 受け入れ基準

- 既存全テスト+新規合格 / typecheck 3構成 / build
- 「わざと雑なコードを書かせる」テストシナリオで、レビューが問題を検出し差し戻し→改善されること
- 実測: ベンチ②(REST API)を reviewGate ON で走らせ、レビュー往復の回数・スコア推移・
  追加コストを docs/autonomy-comparison.md に追記(品質向上とコスト増のトレードオフを記録)
- OFFのユーザーは何も変わらない
