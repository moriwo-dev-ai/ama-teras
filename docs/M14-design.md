# M14 設計: 画像入力対応(UI検証の完成)

**ステータス: 確定(2026-07-06 ユーザー決定済み)**

決定事項:

1. **screenshotの外部URL**: localhostは自動(risk: 'safe')。外部URLは**毎回承認**、ただし
   **「セッション中許可」を選択可**(後述の粒度に注意)
2. **画像保存上限**: セッションあたり**50MB**(超過時は古い画像から「[画像: 説明]」テキストへ置換)
3. **ベンチ④**(Reactアプリ→自分でスクショ→見た目確認→修正、を無介入完遂)を**合格条件に含める**

## 動機

autonomy-comparison.md の最後の主要ギャップ「UI確認(スクショ)不可」を塞ぐ。
Playwright MCP(docs/mcp-starter-pack.md)のテキストスナップショットで**構造**の確認は可能になったが、
「レイアウト崩れ」「色・余白」「ユーザーが貼ったデザインモック通りか」は画像理解が必要。
Claude・GPTとも画像入力はAPIで成熟しており、プロバイダ層の拡張だけで載る。

## スコープ

### M14-1: プロバイダ層の画像対応

- `ChatMessage` のcontentブロックに `image`(base64+mediaType)を追加(`src/shared/types.ts`)
- Anthropic: `{ type: 'image', source: { type: 'base64', ... } }` へ写像
- OpenAI: `image_url`(data URL)へ写像
- **tool_result 内の画像**にも対応(Anthropicはtool_result内imageブロック可。OpenAIは
  tool_resultがテキストのみのため、直後のuserターンに画像を注入する互換レイヤで吸収)
- compaction: 画像ブロックは要約時に「[画像: <説明 or ファイル名>]」へ置換(トークン節約)
- セッション永続化: 画像はbase64のままJSONに入れるとファイルが肥大するため、
  `sessions/blobs/<hash>.bin` に外出しして参照で持つ

### M14-2: 入力経路

1. **ユーザー添付**: チャット欄に画像をドラッグ&ドロップ / ペースト(renderer→IPC→main)。
   「このモックどおりに作って」が可能になる
2. **screenshot ツール**(新規組み込みプラグイン): 指定URLを offscreen BrowserWindow +
   `webContents.capturePage` で開いてスクショ→tool_resultとして画像返却(依存ゼロ・パッケージ版でも動く)
   - **localhost / 127.0.0.1**: risk 'safe' として通常の承認フロー(autoApprove.safeでスルー可)
   - **外部URL**: 毎回承認。ただし**「セッション中許可」を選択可** — 粒度は**ドメイン単位**
     (example.com を許可したらそのセッション中は example.com の再スクショは自動。
     別ドメインは改めて確認)。全部まとめての解放はしない
   - M9のfullPc(allow-session禁止)とはポリシーが異なるが、スクショは読み取りのみで
     破壊的操作がないため緩和してよい、というユーザー決定(2026-07-06)。
     外部URLのスクショは(自動許可されたものも含め)audit.jsonl に記録する
   - file:// や chrome:// 等の非httpスキームは拒否
3. Playwright MCP の screenshot ツールを使う場合も、M14-1のtool_result画像対応でそのまま流れる

### M14-3: UI

- チャット: 画像サムネイル表示(送信前プレビュー・履歴内・tool_result内)
- remote-ui: 同様の表示(スマホからモック写真を撮って送る、が地味に便利なはず)

### M14-4: 検証

- モックプロバイダでのブロック写像テスト、blobs外出しの往復、compaction置換
- 実機E2E: 「Reactでカウンターを作って、devサーバを起動して、スクショを撮って、
  ボタンが中央に見えるか自分で確認して直して」が無介入で通ること(ベンチ④として追加)

### M14-5: fullPcの「セッション中許可」オプション(2026-07-06 追加決定)

M9では systemスコープの allow-session を一律禁止したが、運用してみて毎回承認が煩雑なため緩和オプションを追加:

- `AppConfig.fullPcAllowSession: boolean`(**既定 false** = 従来どおり毎回承認)
- ONのとき、systemスコープの承認ダイアログに「セッション中許可(このフォルダ)」ボタンを表示
  - 粒度は**ツール×ディレクトリ単位**(例: write_file で `C:\temp\foo\a.txt` を許可 →
    そのセッション中は write_file の `C:\temp\foo\` 配下は自動。別フォルダ・別ツールは改めて確認)
  - **exec系(bash等)は対象外**(パスで範囲を限定できないため従来どおり毎回)
  - ハード拒否(userData/システム領域)は設定に関係なく不変
  - セッション許可で自動通過した操作も**全件 audit.jsonl に記録**
- Settings の操作範囲節にトグル+説明(「放置作業の利便と引き換えに、許可したフォルダへの
  書き込みがセッション中ノーチェックになります」)
- テスト: 既定OFFで従来挙動不変(M9回帰テストがそのまま通ること)、ディレクトリ粒度、
  exec除外、audit記録

## 非対象

- 動画・音声
- OCR特化処理(モデルの視覚理解に任せる)
- スクショの差分比較ツール(必要になったら自己進化で)
- UIの3ペイン化(→ docs/M15-ui-design.md として別マイルストーン)

## 見積もり

M11相当(中規模)。プロバイダ写像とセッションblobが本体、UIは薄く。
既存テストへの影響は ChatMessage 型の拡張のみで、後方互換(テキストのみ履歴)は保てる。

## 受け入れ基準

- 既存全テスト+新規テスト合格 / typecheck 3構成
- ユーザー添付画像(D&D/ペースト)がモデルに渡り、内容について応答できる
- screenshot ツール: localhost自動 / 外部は承認(セッション中許可=ドメイン単位)/ 監査記録
- セッションblob 50MB超過時の古い画像テキスト化が動作
- **ベンチ④**: 「Vite+Reactでカウンターを作り、devサーバをbackgroundで起動し、screenshotで
  自分の目で確認して、ボタンが押しやすい見た目になるまで直して」を無介入完遂(合格条件)
