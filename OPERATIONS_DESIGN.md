# Project TAKAMA-gahara 設計(2026-07-11 改訂)

AMA-terasを取り巻く拡張計画の総称。運営自動化はその第1弾であり、今後の機能群もこのプロジェクト名の下に追加していく(命名体系: AMA-teras / TAKAMA-gahara / AMANO-iwate)。

## 現状認識

公開から1日、Star 0 / Fork 0。これは製品の問題ではなく**流入経路が0本**という構造問題(Topicsの新着は数時間で流れる、Zennはいいね初速がないとトレンドに載らない、Xフォロワー0、英語圏未投稿)。

神話になぞらえれば岩戸隠れの状態。岩戸開きの神話では、オモイカネが策を練り、アメノウズメが踊って注目を集め、タヂカラオが戸を開けた。この三神の分業を実装する。

## 大原則(倫理と規約の線引き・変更禁止)

1. **自動化するのは「観測・分析・発見・下書き・提示」まで。外部への発信(投稿・コメント・フォロー・マージ)は人間の承認後にのみ実行する**
2. **各プラットフォームの規約に従う**。規約が自動実行を禁じる媒体(例: Xの自動フォロー・自動エンゲージメント)では、提案止まりとし人間が実行する
3. 自動投稿・自動いいね・サクラ・bot偽装は実装しない(スパムでありブランドの即死要因)

## Protocol: AMANO-iwate(外部媒体アダプタ規約)

天岩戸は内と外の境界。**AMA-terasから外界へ出る通信は、すべてこの岩戸(承認ゲート)を通る。**

### アダプタインターフェース

各媒体を1アダプタとして統一規約で実装する:

```ts
interface MediaAdapter {
  id: string;                    // 'github' | 'zenn' | 'x' | 'bluesky' | ...
  capabilities: {
    read: boolean;               // 観測(メトリクス・投稿・反応の取得)
    search: boolean;             // 発見(候補・言及の検索)
    draft: boolean;              // 下書き生成の対象か
    execute: string[];           // 承認後に実行可能なアクション(空=提案のみ)
  };
  compliance: string;            // 規約上の制約メモ(UIに表示)
}
```

### 共通ルール(岩戸の掟)

- execute系は**全アダプタ共通の承認ダイアログ**(何を・どこへ・全文プレビュー)を必ず通る。承認なし実行はコードレベルで不可能にする
- 全実行を audit 記録。レート制限を尊重。API規約の変更に備え compliance をUI表示
- 新アダプタの追加もこの規約に従う(将来のプラグイン化を見据えた設計)

### 初期アダプタと規約判断

| アダプタ | read/search | execute(承認後) | 規約メモ |
|---|---|---|---|
| github (gh) | ✅ メトリクス/Issue/PR | コメント・ラベル・マージ | API正規利用・問題なし |
| zenn | ✅ 公開API(いいね/コメント数) | なし(提案のみ) | 公開JSON APIの読み取りのみ |
| x | ⚠ 検索URL生成+ユーザー貼り付け解析(セミオート) | **なし(全て人間が実行)** | 自動フォロー/自動投稿は規約違反+新規アカは凍結リスク大 |
| bluesky | ✅ 公開API | 投稿・フォロー(承認後・自動化開示付き) | AT ProtocolはAPI自動化に寛容。Bot開示が礼儀 |
| hatena/reddit/hn | 読み取りのみ | なし(投稿文の下書き+URL提示) | 投稿は人間 |

## 三神の構成

### 🧠 OMOI-kami(オモイカネ/思金神)— 観測と戦略

- 収集: github/zenn/レジストリの各アダプタ経由でメトリクス収集(traffic referrers含む)
- 蓄積: 時系列JSON(userData)。運営タブにトレンド+前回比を表示
- 週報生成: 「何が効いたか(referrer分析)・次の一手」をLLMで下書き
- **メディアポートフォリオの管理**: 各媒体の投下履歴と反応を突き合わせ、効いた経路への集中を提案

### 💃 AMENO-uzume(アメノウズメ/天鈿女命)— 広報と仲間探し

**(a) 発信ドラフト**: PROGRESS.md・コミット・メトリクスから「見せ場」を抽出し、X投稿/リリースノート/記事アウトラインを下書き。コピーboタンのみ、投稿ボタンは作らない

**(b) メディア戦略**: 単一媒体に依存せず、媒体ごとの特性で計画を提示:
- 日本語圏: Zenn(トレンド)→はてブ→X。イベント(connpass)紹介文の下書き
- 英語圏: Show HN / Reddit(r/ClaudeAI, r/LocalLLaMA, r/SideProject)/ Bluesky(AI開発者コミュニティ)
- 中期: デモ動画(YouTube/短尺)、Product Hunt(インストーラ成熟後)

**(c) 仲間発見(コミュニティ・ディスカバリー)**: 「本当にAIを楽しんでいる人(商売目的でない)」を発見し、候補カードとして提示:
- 判定ヒューリスティクス(LLM評価): 自作の作品・コードを公開している / 技術的な失敗談を書く / 有料note・情報商材・LINE誘導などの売り込み導線がない / 宣伝より実験の話が多い
- 候補カード: プロフィール要約+「なぜこの人か」の理由+その人の直近の話題への**返信下書き(宣伝ではなく内容への反応)**
- フォロー・返信の実行は人間(Xの場合)。Blueskyは承認後にアダプタが実行可
- 入力経路: ①検索クエリ・検索URLの自動生成(人間がブラウザで開く) ②見つけたプロフィール/投稿の貼り付け解析 ③Bluesky APIの直接検索

### 💪 TEDIKA-rao(タヂカラオ/天手力男神)— 門番(PR/Issue処理)

- gh経由でIssue/PR取得 → PR diffを既存severity方式レビューにかけ、トリアージカード提示(要旨・指摘・返信下書き・ラベル案・マージ可否推奨)
- 承認後のみghで実行(コメント/ラベル/マージ)、全操作audit
- レジストリのプラグイン投稿PRはCI結果を統合表示

## 実装形態

- 本体の汎用機能として実装(任意の自分のリポジトリ・記事・アカウントを運営できる)。「AMA-terasは自分の運営も自分でやる」がブランド実証
- UI: 右ペイン「運営」タブ(Project TAKAMA-gahara)。三神のセクション+AMANO-iwateのアダプタ状態表示
- コード: `src/main/operations/` 配下に protocol.ts(アダプタ規約+岩戸ゲート)/ adapters/ / omoiKami.ts / amenoUzume.ts / tedikaRao.ts

## v2: 神議(かむはかり)アーキテクチャ(2026-07-11 構想合意)

三神を「ボタンで動く機能」から「時計を持つ自律サブエージェント」へ昇格させ、AMA-terasが統括する3層構造にする。

### 3層構造

1. **神エージェント(下層)**: 時計を持った自律ワーカー。定刻に淡々と仕事(観測・巡回・目利き・下書き)をして、成果を**受け箱(inbox)**に流す。ユーザーとは話さない。安い帯(explorer/worker)で動く
2. **神議 = AMA-teras運営(中層)**: 自分の時計で定期的に開く戦略ループ。受け箱+メトリクス履歴を読み、「どうすれば人を集められるか」をplanner帯で考え、①ユーザーへの**承認バッチ**を作る ②神々への指示調整(キーワード・間隔等のパラメータ変更) ③能力ギャップの検出、を行う
3. **ユーザー(上層)**: まとめられた承認バッチに答えるだけ(バラバラ通知の禁止)

### 神の宣言的定義

神はコードではなく定義データ: `{名前, 時計(間隔), 使うツール群, 判定プロンプト, 出力先, 1日トークン予算}`。
定義がデータであることで、神の新設・改造が「定義の生成+必要なら新ツールの進化」に還元され、既存の検証ゲート+承認のレールに乗る。

### 能力ギャップの3分岐(神議が検出して提案)

- (a) 単発ならAMA-teras本体がその場でカバー(通常のエージェント実行)
- (b) 神への能力追加 = request_capability → 既存の進化パイプライン(検証3段+承認)
- (c) 新しい神の作成 = 新定義の下書き → **進化ジョブと同格の検証+人間承認を必須とする**(組織図の自己改変こそ最も厳格にゲートする)

### ユーザーとの会話場所(UI原則)

- **話す場所**: 左ペイン常設の「⛩ 運営」会話スレッド1本に集約。神議の承認バッチ・分析・相談はすべてここへカードで届き、ユーザーの指示もここ。通常の作業会話と混ぜない。未読バッジ+リモートUI対応
- **見る場所**: 右ペイン運営タブはダッシュボード専用(数字・時計・受け箱)。会話・承認機能は置かない

### 鉄則(v1の大原則に追加)

- スケジューラ(神・神議とも)から外部発信への直通経路は作らない。実行は必ず岩戸ゲート
- 神ごとに1日トークン予算を持ち、超過したら自動で間隔を広げて神議に報告
- 承認依頼は神議がバッチ化して1日N回で提示(既定2回: 朝・夜)
- 神の新設・改造(定義変更)は人間承認必須。定義ファイル群は保護領域に準じた扱い
