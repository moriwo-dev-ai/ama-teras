# 進化レジストリ設計(2026-07-10 協議合意版)

## 基本コンセプト

「作る前に探す」。`request_capability` 発動時、いきなり生成せずレジストリを検索し、既存の進化があれば導入する。生成コストの節約+実戦で揉まれた品質。

## 3層の流通モデル

| 層 | 流通方法 | 理由 |
|---|---|---|
| ツールプラグイン | レジストリで自由流通 | 自己完結モジュール、環境非依存 |
| UIの見た目(テーマ・配色・配置) | 宣言的データ(JSON)として共有 | コードでなくデータなら実行リスクゼロ |
| コア変更・UI機能変更 | 本家リポジトリへのPRで還流 | コア差分は本体バージョン依存で移植不能。公式リリースで全員へ |

## レジストリの実体(フェーズ3で公開)

- **GitHubリポジトリ方式**で開始: プラグイン1つ=1ディレクトリ(コード+テスト+マニフェスト)、`index.json` で索引、登録はPR
- 数百件超で検索がつらくなったら専用APIサーバー(非公開部分)へ昇格

## 導入フロー(検証ゲートは絶対に省略しない)

```
request_capability → レジストリ検索 → ヒット
→「コミュニティに既存の進化があります(作者/DL数/検証済みバッジ)」
→ ダウンロード → ローカルB環境で検証(typecheck→vitest→スモーク)
→ 承認ダイアログ(危険API警告含む) → 導入
```

共有の逆向き: ローカル昇格済みプラグインを「共有しますか?」→ マニフェスト自動生成 → PR化。

## マニフェスト(必須項目)

```jsonc
{
  "name": "...",
  "version": "1.0.0",          // semver
  "pluginApiVersion": "^1",     // 本体との互換範囲
  "description": "...",
  "author": "...",              // クレジット
  "license": "AGPL-3.0",        // AGPL互換必須(投稿時にDCO同意)
  "permissions": {              // 宣言外のAPI使用はCIで自動リジェクト
    "network": false,
    "childProcess": false,
    "fsScope": "workspace"
  },
  "dependencies": []            // フェーズ1は外部依存ゼロをルール化
}
```

## キュレーション体制

- **機械が門番、人間は最終承認だけ**: レジストリCIで typecheck→テスト→静的解析(宣言と実装の不一致検出)→AMA-teras自身のレビューゲート(severity方式)でLLMレビュー→結果をPRに自動コメント。人間はhigh指摘ゼロのPRだけ見る
- 2層掲載: 「検証済み」(全ゲート+人間承認、署名付き)/「未検証」(掲載のみ、デフォルト非表示・オプトイン導入)
- プラグインの所有権・メンテ責任は作者に残し、レジストリは索引と検証のみ担う
- 規模拡大後: 実績あるコントリビューターにプラグイン領域のみのマージ権(コアは渡さない)

## プラグインAPIバージョニング

- プラグインが依存してよいのは `ToolPlugin` インターフェース+渡されるコンテキストのみ。`src/main` 内部のimport禁止(CIで機械検出)。将来 `@ama-teras/plugin-sdk` に切り出し
- semver運用: マイナー=追加のみ、破壊的変更=メジャーのみ。本体は対応範囲(例 `^1`)を宣言
- 範囲外プラグインは**クラッシュではなく無効化+理由表示**で読み込む
- レジストリCIに本体最新N版との互換マトリクステスト

## セキュリティ(多層防御、完全サンドボックスは狙わない)

1. 権限宣言の強制+静的解析(宣言外API使用=自動リジェクト)
2. 外部npm依存ゼロルール(サプライチェーンリスクの最安・最堅の遮断)
3. 検証済みプラグインへのレジストリ鍵署名。本体がダウンロード時に検証
4. ローカルゲート(B環境検証+承認ダイアログ+危険API警告)— 自分のコードも他人のコードも同じ扉を通る
5. **キルスイッチ**: 失効リストを本体が起動時チェック、導入済みでも自動無効化(フェーズ1から実装)
6. サンドボックス実行(別プロセス+Nodeパーミッションモデル)はフェーズ3の検討課題

## 共有フロー(2026-07-11 追記・フェーズ3実装対象)

### 原則: 価値は先に、身元は共有の瞬間だけ

- ツールの生成・利用・レジストリからの導入に**GitHub登録は要求しない**(製品価値の入口に壁を置かない)
- GitHubアカウントが必要になるのは**共有する瞬間のみ**(PRの仕組み上、技術的に不可欠な一点)
- 連携は「GitHubでサインイン」のみ。自前のアカウントDBは作らない(identity は GitHub に一元化)

### 共有の2ルート(併存)

1. **作成時提案**: プラグイン昇格成功のたびに「コミュニティに共有しませんか?」ダイアログを表示。「今後表示しない」で無効化可
2. **手動共有ボタン**: ツール一覧からいつでも共有可能(後から気が変わったルート)

### 共有フロー詳細

```
昇格成功 → 共有提案ダイアログ
→ アップロード内容プレビュー(コード+テスト+マニフェスト)
→ 個人情報スクラブ(絶対パス・秘密情報・固有名の自動除去。除去結果を可視化)
→ 汎用性の事前評価をLLMで実施し「汎用性: 高/低」を表示(ユーザーの判断材料+ジャンクPR抑制)
→ AGPL同意(DCO)チェック
→ 未連携なら GitHub サインイン案内
→ アプリが代行: レジストリをユーザーアカウントへフォーク → コード配置 → 本家へPR作成
→ 「PRを送信しました」
```

### 注意点(設計合意)

- **自動アップロードは絶対にしない**。無断送信はプライバシー事故+DCOは本人の意思表示が必要+「安全」ブランドの即死要因
- **スクラブとプレビューは必須機能**。本人が気づかず個人情報入りコードを共有する事故を防ぐ最後の砦
- 成長指標は「作成数→共有同意率」で計測(エコシステムの健康状態を直接示す)

## 著作権の整理

- AI生成コードの著作権は法的にグレー(人間の創作的寄与の有無で決まる。純自動生成物は著作物でない可能性)
- よって権利の所在に依存せず、**投稿時のライセンス同意(DCO方式)で利用条件を確実に成立させる**
- 権利が存在する範囲では投稿者に帰属、クレジットはマニフェストに保持
- 本格事業化の際は専門家レビューを行う

## 無料APIモードとの関係

- 無料モードのユーザーは**プラグインの消費者**(レジストリからの導入=無料で能力拡張)
- プラグインの新規生成(自己進化)はデフォルト無効(無料モデルは検証ゲート通過率が低く無料枠を溶かすだけになりがち+レート制限で長ループが壊れる)。設定でオプトイン可
- レジストリへの共有は無効(品質はレジストリCIが守るが、フェーズ1はシンプルに倒す)
- 「無料ユーザーが消費し、有料ユーザーが生産する」生態系として設計
